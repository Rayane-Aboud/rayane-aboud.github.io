Implementing a Concurrency-Safe Rate Limiter in Rust

June 2026 · System Design · GitHub repo →

Rate limiting is one of those components that looks trivial on a whiteboard and becomes surprisingly subtle the moment you operate under concurrent load. This post walks through building a production-grade token bucket rate limiter in Rust — from the simplest naive version to a lock-free implementation that holds up under pressure.

Concepts and system requirements inspired by educational materials at DesignGuru. All implementation, code architecture, and analysis here are original and built from scratch for learning purposes.

Overview (5 min)

Before diving into code, here's a short walkthrough of the core ideas — token bucket semantics, the concurrency trap, and how atomic operations save us.

Architecture

The rate limiter sits between the load balancer and your application servers. Every request hits the limiter first; allowed requests flow through, throttled ones return HTTP 429 immediately.

Token Bucket Mechanics

The token bucket algorithm maintains a counter that fills at a fixed rate up to a maximum capacity. Each incoming request consumes one token. If the bucket is empty, the request is rejected.

Two properties define the contract:

  • capacity — maximum burst size (e.g., 100 tokens)
  • refill rate — tokens added per second (e.g., 10 tokens/s)

Naive Implementation (Mutex)

The simplest correct implementation wraps state in a Mutex. It's immediately thread-safe, but every request serializes through a single lock. 

use std::sync::Mutex;
use std::time::{Duration, Instant};

pub struct TokenBucket {
    capacity: u64,
    tokens: u64,
    refill_rate: u64,       // tokens per second
    last_refill: Instant,
}

impl TokenBucket {
    pub fn new(capacity: u64, refill_rate: u64) -> Self {
        Self {
            capacity,
            tokens: capacity,
            refill_rate,
            last_refill: Instant::now(),
        }
    }

    /// Returns true if the request is allowed.
    pub fn allow(&mut self) -> bool {
        self.refill();
        if self.tokens > 0 {
            self.tokens -= 1;
            true
        } else {
            false
        }
    }

    fn refill(&mut self) {
        let elapsed = self.last_refill.elapsed().as_secs();
        let new_tokens = elapsed * self.refill_rate;
        if new_tokens > 0 {
            self.tokens = (self.tokens + new_tokens).min(self.capacity);
            self.last_refill = Instant::now();
        }
    }
}

// Thread-safe wrapper
pub struct RateLimiter(Mutex<TokenBucket>);

impl RateLimiter {
    pub fn allow(&self) -> bool {
        self.0.lock().unwrap().allow()
    }
}

Under load, this becomes a bottleneck. Every call to allow() acquires an exclusive lock — even reads. Under 10,000 RPS on a single key, you'll see the lock contention show up clearly in perf output.

Lock-Free Implementation (Atomics)

The better approach uses a 64-bit atomic to pack both the token count and the last refill timestamp into a single word. Reads and writes become compare-and-swap loops, which the CPU executes without OS involvement. 

use std::sync::atomic::{AtomicU64, Ordering};
use std::time::{SystemTime, UNIX_EPOCH};

pub struct AtomicRateLimiter {
    /// High 32 bits: token count. Low 32 bits: last refill timestamp (seconds).
    state: AtomicU64,
    capacity: u32,
    refill_rate: u32,
}

impl AtomicRateLimiter {
    pub fn new(capacity: u32, refill_rate: u32) -> Self {
        let now = unix_now();
        let initial = pack(capacity, now);
        Self {
            state: AtomicU64::new(initial),
            capacity,
            refill_rate,
        }
    }

    pub fn allow(&self) -> bool {
        let now = unix_now();
        loop {
            let old = self.state.load(Ordering::Acquire);
            let (tokens, last_refill) = unpack(old);

            // Compute refilled tokens
            let elapsed = now.saturating_sub(last_refill);
            let refilled = (elapsed * self.refill_rate as u64)
                .min(self.capacity as u64 - tokens as u64);
            let available = tokens as u64 + refilled;

            if available == 0 {
                return false;
            }

            let new_state = pack((available - 1) as u32, now);
            if self.state
                .compare_exchange(old, new_state, Ordering::AcqRel, Ordering::Acquire)
                .is_ok()
            {
                return true;
            }
            // CAS failed — another thread raced us. Retry.
        }
    }
}

fn unix_now() -> u64 {
    SystemTime::now()
        .duration_since(UNIX_EPOCH)
        .unwrap()
        .as_secs()
}

fn pack(tokens: u32, ts: u64) -> u64 {
    ((tokens as u64) << 32) | (ts & 0xFFFF_FFFF)
}

fn unpack(v: u64) -> (u32, u64) {
    ((v >> 32) as u32, v & 0xFFFF_FFFF)
}

Benchmarks

Measured on an 8-core machine, 10M requests across 64 threads: 

Implementation      Throughput (RPS)   P99 Latency
─────────────────────────────────────────────────
Mutex               ~420,000           12 µs
AtomicU64 CAS       ~3,100,000          2 µs

The atomic version is ~7× faster at the tail and scales linearly with core count. The mutex version saturates around 4 cores — adding more threads makes it slower.
